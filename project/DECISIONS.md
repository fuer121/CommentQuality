## D-004：使用 GitHub 空仓库作为项目远端

日期：2026-05-29

决策：将 `git@github.com:fuer121/CommentQuality.git` 设置为项目远端 `origin`，并以 `main` 作为初始分支。

原因：用户提供了远端仓库地址；本地尚未初始化 Git；`git ls-remote` 未返回 `HEAD/main/master` 引用，按空仓库处理，适合进行项目基线首提交。

影响：基线文件、工作流文件和 `.gitignore` 将作为首个提交推送到远端。后续功能阶段按 `project/GIT_POLICY.md` 的阶段提交策略执行。

## D-006：Dify 节点注入采用线上 API 修改

日期：2026-05-29

决策：Prompt 管理和映射规则管理的注入方式采用直接调用线上 Dify API 修改工作流。

原因：用户明确要求“直接调用 Dify API 修改线上工作流”，并提供了 Dify API 服务器与 API Key。

影响：网站需要把 Dify API Base URL、API Key 等配置放入本地 `.env`，不得提交；实现前必须验证该 API 是否具备工作流编辑能力。若应用 API 只支持运行工作流而不支持编辑节点，总控必须记录阻塞并调整实现策略。

## D-007：Prompt 默认值采用工作流 YAML 节点快照

日期：2026-05-29

决策：书评、章评、段评 Prompt 管理的默认值来自 `dify 工作流/社区评论质量评分-书章段评版.yml` 中对应打分节点的完整 Prompt。

原因：当前 Dify App API 已验证可运行工作流，但 `/workflows`、`/workflows/draft`、`/workflow`、`/apps`、`/app` 等路径返回 404，无法读取或编辑线上节点详情。用户要求默认获取线上节点里的 Prompt，因此在缺少可读节点 API 前，以当前工作流 YAML 作为线上节点基线快照。

影响：后续若拿到 Dify 管理端/Console API 或其他可读写工作流接口，应优先改为实时读取线上节点，并保留 YAML 快照作为离线回退。
