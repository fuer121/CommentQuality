# 决策记录

## D-001：项目真实信源

日期：2026-05-28

决策：使用 `project/` 作为唯一真实信源。所有 Agent 交接、阶段判断、验收证据和经验沉淀必须写入文件。

原因：用户要求不能把线程上下文纳入信源，且后续需要总控回收子 Agent 或独立线程结果。

影响：后续实现前必须先更新对应任务文件或报告文件。

## D-002：当前不自动初始化 Git

日期：2026-05-28

决策：当前目录不是 Git 仓库，基线阶段暂不自动执行 `git init`。

原因：当前目录只有工作流和基线文档，网站工程结构尚未确定。等工程初始化阶段再创建 `.gitignore` 并决定是否建仓更稳。

影响：本阶段只落文件，不提交。进入阶段 1 时重新评估。

## D-003：先不拆子 Agent

日期：2026-05-28

决策：项目基线阶段由总控直接完成，不拆子 Agent。

原因：当前任务是建立控制面和规则，拆分会增加同步成本。实现阶段可拆导入、Dify 调用、前端结果页、测试验证等子任务。

影响：阶段 1 开始前，由总控根据任务边界决定是否拆分。

## D-004：使用 GitHub 空仓库作为项目远端

日期：2026-05-29

决策：将 `git@github.com:fuer121/CommentQuality.git` 设置为项目远端 `origin`，并以 `main` 作为初始分支。

原因：用户提供了远端仓库地址；本地尚未初始化 Git；`git ls-remote` 未返回 `HEAD/main/master` 引用，按空仓库处理，适合进行项目基线首提交。

影响：基线文件、工作流文件和 `.gitignore` 将作为首个提交推送到远端。后续功能阶段按 `project/GIT_POLICY.md` 的阶段提交策略执行。
