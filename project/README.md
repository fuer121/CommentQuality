# 评论质量打分项目基线

本目录是项目唯一真实信源。后续总控、子 Agent、独立线程都必须以 `project/` 下文件为准；聊天上下文只能作为临时沟通，不作为交接依据。

## 当前目标

基于 Dify 工作流 `dify 工作流/社区评论质量评分-书章段评版.yml`，实现一个网站，用于导入社区评论内容并批量跑分，最终得到书评、章评、段评的质量与情绪打标结果。

## 当前已确认输入

工作流输入只允许：

- `comment_type`：评论类型，取值为 `书评`、`章评`、`段评`
- `comment_content`：评论内容，纯文字

## 当前已确认输出

网站和服务层必须围绕以下字段组织结果：

```json
{
  "comment_type": "书评、章评、段评",
  "quality_score": "质量分数（0-100）",
  "quality_level": "质量等级（好、中、差根据quality_level映射）",
  "quality_reason": "评分理由，需说明质量等级和主要原因",
  "emotion_score": "情绪分数（0-100）",
  "emotion_type": "负向、中性、正向"
}
```

## 非目标

- 不做图片、表情包、视频、链接卡片质量评估。
- 不把聊天记录作为需求来源或 Agent 交接材料。
- 不在没有验收证据时声称阶段完成。

## 总控原则

- 先更新本目录文件，再启动实现或子 Agent。
- 每个阶段必须有明确验收标准。
- 子 Agent 只能写入 `project/agent-reports/` 或其被分配的任务文件，不直接改动基线结论。
- 总控回收子 Agent 结果后，必须做判断：进入下一阶段、要求返工、或调整任务拆分。
