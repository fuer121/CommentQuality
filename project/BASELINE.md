# 需求与实现基线

## 背景

社区已有帖子质量判断能力，现在需要面向书评、章评、段评建立评论质量与情绪打标能力，并提供网站完成评论内容导入、批量跑分、结果查看与导出。

## 工作流基线

目标工作流文件：`dify 工作流/社区评论质量评分-书章段评版.yml`

工作流设计要点：

- 输入只包含 `comment_type` 和 `comment_content`。
- `comment_type` 限定为 `书评`、`章评`、`段评`。
- 书评、章评、段评分别使用独立 Prompt。
- `quality_score -> quality_level` 和 `emotion_score -> emotion_type` 由可配置映射节点生成。
- 不启用图片或视觉模型。

默认映射关系：

| 字段 | 分数区间 | 标签 |
|-|-|-|
| `quality_level` | `[80,100]` | 好 |
| `quality_level` | `[30,80)` | 中 |
| `quality_level` | `[0,30)` | 差 |
| `emotion_type` | `[80,100]` | 正向 |
| `emotion_type` | `[30,80)` | 中性 |
| `emotion_type` | `[0,30)` | 负向 |

## 网站能力基线

MVP 必须支持：

- 导入评论数据。
- 校验每条数据是否包含评论类型和评论内容。
- 批量调用目标 Dify 工作流跑分。
- 展示每条评论的评分结果、运行状态和错误原因。
- 支持结果导出。
- 保存一次批量任务的输入、输出和运行日志，便于复盘。

## 导入数据最小字段

导入文件至少需要以下列：

- `comment_type`
- `comment_content`

允许中文列名映射：

- `评论类型` -> `comment_type`
- `评论内容` -> `comment_content`

## 初始技术假设

当前目录还不是 Git 仓库，也没有网站工程。实现阶段需要先确认或创建网站工程结构。

优先方案：在当前目录内创建一个轻量 Web 应用，包含前端页面、服务端 Dify 调用代理、批量任务存储和导出能力。

## 未决问题

- Dify API Base URL 和 API Key 需要在实现阶段通过 `.env` 配置，不写入仓库。
- 导入格式优先支持 CSV 还是 Excel，需要在实现任务中根据本地依赖和用户常用场景确定；MVP 可同时支持 CSV 和 XLSX，如果成本过高先支持 CSV。
- 是否需要鉴权、多人协作、历史任务持久化到数据库，暂不纳入 MVP。
