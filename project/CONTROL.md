# 总控协议

## 唯一真实信源

`project/` 是唯一真实信源。任何任务、结论、验收、风险、经验沉淀，都必须写入文件系统后才算被项目接收。

聊天上下文只用于即时协作，不作为后续 Agent 或线程的输入依据。

## 总控职责

总控负责：

- 拆分任务和定义阶段门槛。
- 判断是否需要子 Agent 或独立线程。
- 分配任务文件和回收报告。
- 审核子 Agent 产出是否满足验收标准。
- 决定是否进入下一阶段、返工、缩小范围或扩展范围。
- 决定 Git 提交、推送、PR 的时机。
- 维护经验沉淀，避免重复踩坑。

## 子 Agent 使用规则

只有满足以下任一条件时才拆子 Agent：

- 任务可以并行推进，且边界清楚。
- 需要独立调研或验证，不应阻塞主线实现。
- 需要长时间运行或在独立线程持续跟进。
- 任务产物可通过文件明确交付和验收。

不拆子 Agent 的情况：

- 单文件或小范围修改。
- 任务依赖尚未明确。
- 验收标准还没有写清楚。
- 子任务之间强耦合，拆开会增加同步成本。

## 子 Agent 交付要求

子 Agent 必须提交报告到 `project/agent-reports/`，报告至少包含：

- 任务 ID
- 输入文件
- 变更文件
- 验证命令和结果
- 未解决问题
- 建议下一步

总控不能直接接受“已完成”的口头结论，必须依据文件、命令输出或可复现证据判断。

## 阶段门槛

每个阶段默认需要满足：

- 需求边界已写入 `project/BASELINE.md` 或任务文件。
- 实现变更已落盘。
- 验证命令已运行，结果已记录。
- 风险和未决问题已记录。
- 总控明确给出进入下一阶段的判断。
