# 经验沉淀

## 当前项目经验

- Dify 工作流文件必须作为实现契约读取，不要只凭口头描述接入字段。
- 评论质量打标当前只处理纯文字，不要引入图片、表情包、视频、链接卡片评估。
- `quality_level` 和 `emotion_type` 是映射结果，不应让前端或服务端各自重复定义一套不一致规则。
- 映射默认规则来自飞书参考文档：质量 `[80,100]=好`、`[30,80)=中`、`[0,30)=差`；情绪 `[80,100]=正向`、`[30,80)=中性`、`[0,30)=负向`。
- 当前目录不是 Git 仓库；进入实现阶段前要先建立 `.gitignore`，避免 `.DS_Store`、`.env`、导入数据和导出结果进入版本库。
- 2026-05-29 已将空远端 `git@github.com:fuer121/CommentQuality.git` 初始化为 `origin`，首个基线提交为 `5c9b9a6 chore: establish project baseline`，`main` 已跟踪 `origin/main`。
- Dify App API 可运行 `/workflows/run`，但当前未暴露工作流节点读写端点；Prompt 默认值先以目标 YAML 中对应节点的完整 Prompt 为准，不能用前端/后端临时压缩文案冒充线上节点默认值。
- 浏览器侧发现中文 Excel 文件名乱码时，根因是上传文件名被按 latin1 解释；服务端需要在保存任务前对疑似 mojibake 的 `originalname` 做 UTF-8 还原。
- 配置默认值升级要考虑已有 `data/config.json`：如果用户保存过早期简化默认 Prompt，读取配置时应自动迁移到新的工作流节点完整 Prompt，同时保留用户自定义 Prompt。
- 当前批量跑分是阻塞式顺序调用 Dify；任务级暂停只能保证“当前单条完成后不再跑下一条”，不能取消已经发出的 Dify 请求。
- 任务历史上限应在服务端 `readTasks/writeTasks/upsertTask` 统一裁剪，前端只负责滚动展示，避免 UI 和存储保留策略不一致。
- 运行中任务的列表进度必须在每条评论完成/失败后立即更新 `successRows/failedRows`，并在读取任务时按行状态校正旧摘要，否则明细已完成但列表会继续显示 `0/总数`。
- 没有人工标注字段时，不能声称质量/情绪评分准确率，只能输出无监督审计结论和待标注样本。
- 纯文字评论评分 Prompt 不能要求模型判断“贴合原文/章节/段落氛围”，因为当前输入只有评论类型和评论内容；评分理由必须只引用评论文本中的可见证据。
- 段评短泛评是当前 V1 的主要误放来源；`爽`、`哈哈`、`来了`、`快`、`牛逼` 等无对象短评应默认设置质量上限，避免把互动感当成高质量。
- V2 Prompt 可保留为本地 Prompt 管理和后续灰度参考；当前新线上工作流跑分固定发送 `prompt_version=V1`，不能再根据前端配置向 Dify 输入 V2
- 前端长任务轮询只应刷新任务状态，不应持续刷新配置接口，否则会覆盖用户正在编辑但尚未保存的 Prompt 和映射配置。
- 本地 JSON 存储在长任务频繁写入、前端高频轮询读取时必须原子写入；读取 JSON 解析失败不能静默 fallback 成空数据，否则页面会误清空任务列表和详情。
- 新线上 Dify 工作流运行入参以 `/parameters` 为准：评论类型变量为 `type`，书评/章评/段评分别传 `1/2/3`；评论内容变量为 `content`；版本变量为 `prompt_version` 且固定 `V1`；测试标记变量为 `is_test` 且默认 `0`
- 新线上 Dify 工作流真实输出是紧凑结构 `outputs.result.result/reason/emotion_score/version`，网站需要本地归一化出 `comment_type/quality_level/emotion_type`，不要假设工作流一定直接返回六字段契约
- Prompt 自循环优化应以最新完成任务为主样本源；如果最新任务仍在 running，脚本应停止等待，避免基于半截数据固化 Prompt
- 书评、章评、段评不能共用同一评分标准：书评重完整阅读评价和推荐价值，章评重章节讨论和角色/剧情反馈，段评重短文本中的对象、判断、梗点和信息增量
- V3 Prompt 当前是离线候选，可用 `npm run build:prompt-self-loop -- --task-id <任务ID>` 复跑生成自循环分析产物，不直接覆盖线上工作流

## 复用规则

每完成一个阶段，总控必须更新本文件，至少记录：

- 本阶段踩过的坑。
- 有效的验证命令。
- 需要后续 Agent 复用的路径、字段、接口或配置。
- 任何不能再从 0 重新判断的结论。
